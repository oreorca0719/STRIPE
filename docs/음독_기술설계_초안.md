## §2. 기술 설계

### 2-1. 테이블 — `oral_reading_results` 신설 (명세 준수)

현재 코드는 음독·묵독을 `fluency_results` 한 테이블에 `type` 컬럼으로 통합했다.
음독을 여기에 얹지 않고 **명세대로 별도 테이블을 신설**한다.

**근거:** 음독 13필드 중 6개(`stt_*`, `transcript_length_ratio`, `audio_stored`)가
STT 전용이다. 통합하면 묵독 행에서 전부 NULL이 되고, 어느 컬럼이 어느 모드용인지
스키마만 봐서는 알 수 없게 된다. 명세와 구조를 맞춰두면 이후 대조 검증도 쉽다.

`fluency_results`는 묵독 전용으로 남기되, 기존 데이터·코드는 건드리지 않는다.

### 2-2. 오류 분류 — 정렬(alignment) 기반

기존 `analyzer.py`는 **음절 수 차이를 오류로 근사**한다. 이 방식은 **대치를 잡지
못한다** — 틀리게 읽어도 음절 수는 같기 때문이다. 즉 현재 정확성 지표는 사실상
"누락률"만 재고 있다. 재작성한다.

**알고리즘:** 원문과 전사문을 **음절 단위로 정렬**(Needleman–Wunsch 계열)한 뒤,
정렬 결과에서 오류 유형을 판별한다.

| 유형 | 정렬 패턴 | 판별 |
|---|---|---|
| 대치 (substitution) | 원문 음절 ↔ 다른 음절 | 정렬에서 직접 |
| 생략 (omission) | 원문 음절 ↔ gap | 정렬에서 직접 |
| 첨가 (insertion) | gap ↔ 전사 음절 | 정렬에서 직접 |
| 반복 (repetition) | 전사에 동일 음절·어절 연속 중복 | **후처리 패턴 매칭** |
| 수정 (self-correction) | 오류 직후 정확 발화가 따라옴 | **후처리 패턴 매칭** |

산출:
```
정확 음절 수 = 전체 음절 - (대치 + 생략)
A1_CWPM      = 정확 음절 수 ÷ (A1_total_time_ms / 60000)
A2_accuracy  = 1 - (5종 오류 합 ÷ 전체 음절)
```

### 2-3. ⚠️ 이 설계의 근본 한계 — 반복·수정은 STT로 못 잡을 수 있다

**상용 STT는 정제된 문장을 출력한다.** 말더듬·반복·자기교정을 발화 그대로 전사하지
않고 지워버리는 경우가 많다. 그러면 5종 중 **반복·수정 2종이 구조적으로 측정
불가능**해진다.

이건 우회 설계로 해결되지 않는다. 확인해야 할 사실이고, **Phase 1 벤치마크의 1순위
검증 항목**이다. 결과에 따라 셋 중 하나를 택하게 된다.

1. 반복·수정이 전사에 남는다 → 5종 전부 자동 분류 (A안 성립)
2. 남지 않는다 → **3종(대치·생략·첨가)만 자동**, 나머지는 측정 제외로 명시
3. 정확도 자체가 낮다 → **B안(하이브리드)** 전환. 타이머만 자동, 오류는 감독자 입력

**② 를 택할 경우 `A2_accuracy`의 정의가 명세와 달라진다**(5종 합 → 3종 합).
기획 합의가 필요한 사항이다.

### 2-4. STT 품질 게이트

묵독의 A4 타당성 범위(0.3~15.0 음절/초)와 같은 취지의 방어를 음독에도 둔다.
명세가 `stt_quality_flag`·`transcript_length_ratio`를 미리 넣어둔 것은 STT가 못
알아들을 수 있음을 전제한 설계로 읽는다.

```
transcript_length_ratio = 전사 음절 수 ÷ 원문 음절 수

fail  → 판정에서 제외, 재측정 유도
low   → 판정에 사용하되 reliability_flag = low
pass  → 정상
```

판정 입력은 `stt_confidence`와 `transcript_length_ratio` 두 값이다. 구체적 임계값은
파일럿 데이터로 정한다(§5-3, §5-4). **잠정값으로 시작하되 결과 화면에 "잠정 기준"
표기를 유지**한다 — 묵독과 같은 원칙이다.

### 2-5. 음성 파일 처리

아동 음성은 민감정보다. 다음을 기본 설계로 둔다.

- **기본은 미저장.** `audio_stored=false`, 전사 후 즉시 폐기
- 저장이 필요한 경우(오류 분류 검증·이의 제기 대응)에만 명시적으로 켠다
- 저장 시 보관 위치·기간·파기 절차는 STR-86 방침에 반영해야 한다
- 보호자 동의서에 **음성 녹음 별도 고지** 필요 — 명세 §1-2 `consents.audio_consent`
  필드가 이미 이를 전제하고 있다

### 2-6. 진단 흐름 통합

```
SCR-06 마이크 점검 (mic_match_ratio 확인)
   ↓
SCR-08 음독 진행 — 녹음 → 업로드 → STT → 분석 → oral_reading_results 저장
   ↓
§3-1 유창성 판정에 oral 분기 추가
   fluency_source = "oral", fluency_value_unit = "CWPM"
```

명세 §3-1은 **silent 경로 의사코드만** 있다. oral 분기와, 초4 병행 모드에서
음독·묵독 값이 모두 있을 때의 우선순위는 기획 결정이 필요하다(§5-6).

## §3. 검증·테스트 계획

명세에 테스트 방식이 없어 여기서 정의한다.

### 3-1. Phase 1 — STT 정확도 벤치마크 (착수 조건)

**목적:** A안(STT 자동)이 성립하는지 판정. 여기서 실패하면 이후 설계가 전부 바뀐다.

**측정 항목**
| 항목 | 왜 |
|---|---|
| 음절 단위 일치율 | 기본 정확도 |
| **반복·자기교정 보존 여부** | §2-3 — 5종 분류 가능성 판정 |
| `stt_confidence` 분포 | 품질 게이트 임계값 근거 |
| `transcript_length_ratio` 분포 | 게이트 범위 근거 |
| 학년별·성별 편차 | 특정 집단만 인식률이 낮으면 측정 형평성 문제 |

**표본:** 아동 실제 발화가 원칙. 확보 전에는 성인 낭독으로 대리 측정하되,
**아동 대비 과대평가된 수치임을 명시**한다. 아동 음성 인식률은 성인보다 낮다.

**합격 기준은 기획 결정 사항**(§5-7). 기준 없이 측정만 하면 판단할 수 없다.

### 3-2. 단위 테스트 — 오류 분류기

정답이 있는 합성 케이스로 검증한다. 실제 음성 없이 가능하다.

```
원문 "학교에 갔다"  전사 "학교에 갔다"      → 오류 0
원문 "학교에 갔다"  전사 "학교에 왔다"      → 대치 1
원문 "학교에 갔다"  전사 "학교 갔다"        → 생략 1
원문 "학교에 갔다"  전사 "학교에 다 갔다"   → 첨가 1
원문 "학교에 갔다"  전사 "학교 학교에 갔다" → 반복 1
원문 "학교에 갔다"  전사 "학고 학교에 갔다" → 수정 1
```

경계 케이스: 전사 공백(무음), 원문보다 훨씬 긴 전사(잡음), 전사 실패.

### 3-3. 통합 테스트

진단 테이블은 JSONB를 쓰므로 SQLite로 만들 수 없다. **실 Postgres 통합 테스트**로
둔다(`STRIPE_IT=1`). CI 통합 스텝은 `pytest tests/` 전체를 돌리므로 자동 포함된다.

### 3-4. 회귀 방어

음독 추가가 **묵독 판정을 바꾸면 안 된다.** `fluency_source=silent` 경로의 기존
테스트가 그대로 통과하는지 매 단계 확인한다.

## §4. 작업 순서

```
Phase 0  [기획]  §5 결정 대기 항목 확정
Phase 1  [검증]  STT 벤치마크 — A/B/C안 판정          ← 여기서 갈림
Phase 2  [모델]  oral_reading_results 마이그레이션
Phase 3  [엔진]  정렬 기반 오류 분류 + A1/A2 산출 + 품질 게이트
Phase 4  [화면]  SCR-06 마이크 점검 + SCR-08 음독 진행
Phase 5  [통합]  §3-1 oral 분기 + diagnosis_mode 병행 처리
```

**Phase 1이 Phase 2~5보다 먼저인 이유:** 아동 STT 정확도가 낮으면 A안이 성립하지
않고, B안(하이브리드)으로 가면 **분석기와 화면 설계가 통째로 달라진다.** 순서를
바꾸면 만들고 버리게 된다.

Phase 1은 측정 방식 결정(STR-16) 없이도 착수 가능하다. 오히려 **이 숫자가 있어야
STR-16 결정이 근거를 갖는다.**

## §5. 기획 결정 대기 항목

| # | 항목 | 영향 |
|---|---|---|
| 1 | `error_types` **유형 목록 확정** | 도메인 문서는 생략·대치·첨가·반복·수정. STR-83 이슈는 삽입·자기교정으로 표기 — 같은 것이나 명칭 통일 필요 |

| 3 | `stt_quality_flag` 판정 기준 | confidence·ratio 임계값 |
| 4 | `transcript_length_ratio` 허용 범위 | 미독·중단 탐지 |

- **`POST /api/audio/oral`에 인증 가드 없음.** 토큰 없이 호출 가능하다. 현재는
  `CLOVA_API_KEY`가 비어 Mock으로 떨어져 실피해가 없으나, **키를 넣는 순간

- **`analyzer.py`는 재작성 대상**(§2-2). 대치를 못 잡아 정확성 지표가 실제로는
  누락률만 재고 있다.
- `MVP1_갭분석_v1.2대비_현재코드.md`는 2026-06-06 스냅샷이라 현재와 크게 다르다.
  거기서 🔴로 표시한 테이블·엔진은 대부분 구현 완료. **⚪ 구조보존 영역(음독·부모/
  교사·followup) 서술만 아직 유효하다.**
